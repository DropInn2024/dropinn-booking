## 第一部分：已完成的除錯修正（請 Cursor 先確認）

### 修正 1：adminGetAllOrders() 強制序列化（main.js）

**根本原因：**
google.script.run 傳回前端時，若資料含非標準型別
（空字串混入數字欄位如 extraBeds/paidDeposit、boolean 如 reminderSent），
GAS 序列化失敗直接回傳 null，導致前端 Array.isArray(data) === false，
allOrders 永遠是空陣列，表格顯示「共 0 筆」。

**修正後的 adminGetAllOrders()（main.js）必須是：**
```javascript
function adminGetAllOrders() {
  var orders = DataStore.getOrders();
  return JSON.parse(JSON.stringify(orders)); // 強制清理，避免 GAS 傳 null
}
```
請確認此函式有 JSON.parse(JSON.stringify(orders)) 這行。

## 第二部分：通知與 Email 流程（已實作）

- **待確認**：客人下單後寄「待確認信」— 48 小時內確認、住宿須知連結、聯絡 LINE / IG / 電話 0967-212-168（純文字）。建立訂單時由 `EmailService.sendPendingConfirmationEmail` 發送；管理員通知由 `sendNewOrderNotification` 發送。
- **預定中**：狀態改為預定中後寄「確認信」— 總額、訂金、尾款、代訂機車／行程說明、聯絡方式。在 `updateOrderAndSyncInternal` 中當狀態變為預定中時呼叫 `EmailService.sendConfirmationEmail` 與 `sendAdminStatusNotification`。
- **入住前 7 天**：由 `travelGuideSender.js` 的 `checkAndSendTravelGuides()`（每日觸發）自動寄旅遊手冊，內容以連結為主。設定觸發器：`setupTravelGuideTrigger()`。
- **已取消（無訂金）**：寄「感謝信」— 開心口吻、邀請再來、聯絡方式。`EmailService.sendCancelEmail` 依 `paidDeposit` 判斷。
- **已取消（有訂金）**：寄「退訂＋確認退款信」— 註明退款金額、請確認是否入帳、聯絡方式。
- **每次狀態變更**：皆寄管理員通知信（訂單摘要＋可複製的 LINE 文案），`EmailService.sendAdminStatusNotification(order, status)`。

## 第三部分：折扣碼與成本表規格（已實作）

### 訂單欄位（schemaManager.js）
- `originalTotal`：折扣前原價  
- `totalPrice`：折後總價  
- `paidDeposit`、`remainingBalance`  
- `discountCode`、`discountType`、`discountValue`、`discountAmount`  
- `isReturningGuest`、`complimentaryNote`（老客人／招待備註）  
- **成本與代訂**：`agencyName`（同業來源，空白＝直客）、`addonAmount`（代訂代收金額，行程／租車等，非收入）、`extraIncome`（其他收入，業者退給民宿等）。  

### 訂單表 Schema 更新：既有試算表如何處理（不必整張重建）
- 程式是依「**欄位順序**」讀取（第 1 欄＝orderID、第 2 欄＝createdAt…），若試算表是舊版、少了 `agencyName`／`addonAmount`／`extraIncome` 三欄，後面欄位會錯位，就會讀不到之前的訂單。
- **做法一（推薦）：在 GAS 執行 Sheet 初始化，會自動補齊欄位。**
  - 在 GAS 指令碼編輯器選函式 **initializeYearSheet(年份)**（來自 `setup.js`）→ 執行。**不傳參數＝今年**，或傳入數字如 `2024` 指定年份。
  - 若該年工作表**已存在**，會自動在 **complimentaryNote** 右側插入 3 欄並寫入標題，既有資料不變。（不需另跑「只補欄」函式，同上即可。）
- **做法二（手動）：** 在試算表「訂單_YYYY」中，於 **complimentaryNote** 右邊插入 3 欄，第一列依序填 `agencyName`、`addonAmount`、`extraIncome`。多個年份表要各做一次。
- 完成後舊訂單會正常被讀取，新三欄可留空，之後在後台訂單詳情「成本與代訂」區塊填寫即可。

### 折扣碼
- 試算表工作表「折扣碼」：程式會在第一次使用時自動建立，欄位 `code`, `type`, `value`, `description`, `useLimit`, `usedCount`, `validFrom`, `validTo`，第 2 列會自動填入使用說明備註。  
- API `checkCoupon(code, originalTotal)`：檢查使用次數、有效期限，回傳 `valid`, `discountAmount`, `discountType`, `discountValue`。  
- 類型支援：`fixed`（固定金額）、`percent`（百分比）。  
- 前台：index.html 優惠碼輸入與套用，送出時帶 `originalTotal`, `totalPrice`, `discountCode`, `discountAmount` 等。  
- 後台：訂單詳情可手動改「折後總價」，尾款即時重算。

### 成本表（支出_YYYY）
- 欄位：`orderID`, `name`, `checkIn`, `rebateAmount`, `complimentaryAmount`, `otherCost`, `note`。  
- 建立訂單時 `DataStore.appendCostRow(orderID, name, checkIn)` 自動新增一列。  
- **後台寫回**：在訂單詳情「成本與代訂」區塊填寫後儲存，`updateOrderAndSync` 會將退佣／招待／其他支出／備註寫回成本表該列（`DataStore.updateCostRowByOrderID`）。  
- 訂單狀態改為「已取消」時，`DataStore.clearCostRowForOrder(orderID, year)` 將該列支出清為 0。

### 同業標記與退佣統計
- 同業來源存於訂單欄位 `agencyName`，**退佣（支出）**存於成本表 `rebateAmount`。  
- 詳細報表會依 `agencyName` 彙總「民宿要給各同業多少」退佣（`byAgency`）。直客為 `agencyName` 空白時顯示為「直客」。

### 報表項目定義（收入／支出，避免搞錯方向）

| 項目 | 定義 | 欄位／備註 |
|------|------|------------|
| **房間營收（折後）** | 住宿費折後總額，民宿實收的房價 | 訂單 `totalPrice` |
| **代訂代收金額** | 客人付給民宿的「代訂」總額（行程、租車等），**不是收入**，只是代收；尚未扣掉要給業者的錢 | 訂單 `addonAmount` |
| **退佣（支出）** | **民宿給業者／同業的錢**（我們付出去的） | 成本表 `rebateAmount` |
| **業者給民宿的退佣／回饋** | 業者（機車行、旅行社等）退給民宿的錢 → **算收入** | 訂單 `extraIncome`，報表上歸在「其他收入」 |
| **招待費用** | 民宿自己的支出（招待客人等） | 成本表 `complimentaryAmount` |
| **其他支出** | 其他成本 | 成本表 `otherCost` |
| **其他收入** | 業者給民宿的退佣、回饋等 | 訂單 `extraIncome` |
| **淨利** | 房間營收 ＋ 其他收入 － 退佣（支出）－ 招待 － 其他支出（**不含代訂代收**，因代收要再付給業者） | 計算欄 |

- **後台總覽**：只顯示精簡項目（房間營收、代訂代收金額、退佣＋招待＋其他成本、其他收入、淨利、訂單數），且可選「月份」。
- **詳細報表**：比後台更細，含已收訂金、剩餘尾款、折扣碼總折抵、老客人訂單數、分月、同業退佣統計、訂單明細、並可「產生報表」下載。

## 第四部分：財務報表（已實作）

- GAS：`getFinanceStatsInternal(year, month)`（`month` 可選，0 或省略＝全年，1–12＝該月入住）。

## 第六部分：同業帳號與日曆系統（規劃）

同業可自助註冊、自訂同業名稱與民宿名稱；一棟民宿一個日曆，點日期＝標記「無法提供包棟」。資料僅記錄「有無房可接包棟」，不存同業訂單金額等隱密資訊；推薦成功可備註在系統日曆（退佣多數為服務、無退佣也可註記）。

### 三張 Sheet（用戶執行 GAS 腳本建立）

| 工作表 | 用途 | 欄位（建議） |
|--------|------|--------------|
| **AgencyAccounts** | 同業帳號 | `agencyId`, `loginId`, `passwordHash`, `displayName`, `createdAt`, `isActive` |
| **AgencyProperties** | 同業名下民宿／棟別 | `propertyId`, `agencyId`, `propertyName`, `sortOrder`（一棟一日曆） |
| **AgencyBlocks** | 無法提供包棟的日期 | `propertyId`, `date`（YYYY-MM-DD）, `createdAt` |

- 一棟民宿一個 property，同業可有多個 property（如 A 棟／B 棟）。
- 內建一組管理員測試帳密（如 test_agency / DropinnTest2026!），其餘由業者自行註冊；後台可重設密碼。

### API 規格（GAS doGet/doPost）

- **註冊**：`action=agencyRegister`，body: `loginId`, `password`, `displayName` → 寫入 AgencyAccounts、回傳 token 或 session。
- **登入**：`action=agencyLogin`，body: `loginId`, `password` → 驗證後回傳 token 或 session。
- **取得我的民宿列表**：`action=agencyGetProperties`，需同業 token → 回傳該 agency 的 AgencyProperties。
- **取得／寫入 blocked 日期**：`action=agencyGetBlocks` / `action=agencySetBlock`，需 propertyId + token；Set 傳 `date` 或 `dates[]` 新增／移除。

### 同業端頁面（agency.html）

- **註冊／登入**：表單送出後呼叫上述 API，登入成功後存 token（或 session cookie），進入日曆區。
- **日曆區**：可切換 A 棟／B 棟（依 AgencyProperties 列表），點日期＝標記該日「無法提供包棟」或取消標記；另有一個「總覽」日曆（只讀），以淡色區分各棟滿房狀況。
- 部署方式與現有 GAS 一致（例如 HtmlService 依參數輸出 agency.html 或由靜態託管提供）。

### 後台（admin）

- 新增「同業日曆」區塊：可選同業＋棟別，查看該同業的 blocked 日期；跨同業總覽可判斷「某天哪間還能訂」。
- 管理員可重設同業密碼（呼叫 GAS API 更新 AgencyAccounts）。

## 第七部分：公開空房查詢頁（availability.html）

- 靜態頁 `availability.html` 部署在 GitHub Pages，載入 `config.public.js`，以 `FRONTEND_CONFIG.API_URL_PUBLIC` 呼叫 GAS。  
- 使用 `doGet?action=getBookedDates`，回傳格式：  
  - `success: true/false`  
  - `booked: string[]` — 狀態為「預定中」訂單展開的所有日期（入住日至退房日前一日，格式 `YYYY-MM-DD`）  
  - `pending: string[]` — 狀態為「待確認」訂單展開的所有日期。  
- 前端 `availability.html` 以兩個 `Set`（`bookedDates` / `pendingDates`）標記月曆格子顏色：空房／待確認／已預定；月份導覽由 `moveMonth(delta)` 控制。  
- 初始預設日期僅用於開發預覽，實際上 `fetchStatus()` 會在成功取回 API 後覆蓋。  
- 若日後變更 API 路徑，只需更新 `config.public.js` 的 `API_URL_PUBLIC`。

### 二、Schema 版本更新策略（減少手動增刪欄）

- **單一來源**：訂單表欄位只定義在 `schemaManager.getSchema()`／`getHeaders()`。
- **建表**：新建「訂單_YYYY」時一律用 `SchemaManager.getHeaders()` 寫入第一列。
- **既有表**：`DataStore.ensureOrderSheetSchema(sheet)` 檢查是否缺欄（目前實作：缺 `agencyName` 則在 `complimentaryNote` 右側插入 3 欄）。未來若再新增欄位，只需在 schemaManager 加定義，並在 `ensureOrderSheetSchema` 內加一組「若缺某欄則插入」的邏輯，不必再手動改試算表。
- **GAS 執行**：跑 `initializeYearSheet()`（今年）或 `initializeYearSheet(2024)`（指定年）即會建表或補欄；`setupSystem()` 每次也會跑一次今年表初始化，等於自動做版本更新。

### 三、單一設定頁構想（一頁完成檢查、建表、觸發器、帳號申請）

目標：後台一個「系統設定」頁，可完成大部分初始化與檢查，減少到 GAS 編輯器選函式執行的次數。

| 功能 | 做法 | 備註 |
|------|------|------|
| **檢查狀態** | 後台呼叫 `adminQuickCheck()`（或新 API）回傳 Properties／日曆／試算表／觸發器狀態 | 唯讀，可從 doGet/doPost 呼叫。 |
| **建立／補齊今年表** | 後台按鈕 → `adminInitializeYearSheet(year)`（現有或新增 admin API） | 已有 DataStore 邏輯，僅需對外 API。 |
| **一鍵日常初始化** | 後台按鈕 → 呼叫 `setupSystem()` 的邏輯（或包成 `adminRunSetupSystem()`） | 需在 main.js 新增 action，內部呼叫 setup 的步驟。 |
| **設定 Email／提醒／旅遊手冊觸發器** | 後台按鈕 → 呼叫 `setupEmailTriggers()`、`setupReminderTrigger()`、`setupTravelGuideTrigger()` | GAS 允許在「以部署者身份執行」的 Web App 中建立觸發器，可由後台觸發。 |
| **帳號申請／使用說明** | 同一頁靜態區塊：如何申請 reCAPTCHA、如何取得試算表 ID、如何第一次部署、Script Properties 清單 | 不呼叫 API，僅文件與連結。 |

實作時建議：

1. 在 **main.js** 新增 admin-only actions：例如 `runSetupSystem`、`initializeYearSheet`、`runSetupEmailTriggers`、`runSetupReminderTrigger`、`runSetupTravelGuideTrigger`、`quickCheck`（回傳 JSON 狀態）。
2. 在 **admin.html** 新增「系統設定」分頁或區塊：顯示檢查結果、按鈕對應上述 API，加上「帳號申請與使用說明」摺疊區。
3. 仍保留在 GAS 編輯器手動執行 **setupEverything()** 的選項（第一次建立新日曆、寫入 Properties 範本），因涉及建立日曆與敏感設定，可僅文件說明不強制做進後台。

這樣可達到：**日常與版本更新以「後台一頁＋自動補欄」為主，必要時再進 GAS 執行一次性設定（如 setupEverything）**。

## 未來功能備註：同業串接訂單

**構想**：同業（旅行社、機車行等）可在**自家管道**直接下訂（例如嵌入的日曆／表單、專屬連結），系統自動辨識為「同業單」，由對方**自行選擇或輸入業者名稱**，訂單建立時自動帶入 `agencyName`，後台報表與退佣統計即可依同業彙總。

- **效益**：減少民宿手動標記同業、同業不用透過一般訂房頁或 LINE 傳資料，流程更短、資料一致。
- **實作方向**：專屬訂單入口（如 `?source=agency` 或獨立頁）、表單多一欄「業者名稱」（下拉或文字），建立訂單時寫入 `agencyName`；若有白名單 token 可一併帶入，方便區分來源。日後可再擴充「同業專屬日曆」或 API 供對方系統串接。

## 包棟訂房：日曆上勾選（建議優先）

**現狀**：目前前台是「表單先填入住／退房日期」（date 輸入框），較適合多房型、多間數的複雜選擇；包棟本質上就是選一段連續日期，最直覺是**在日曆上直接勾選**。

**建議流程**：
1. **第一步：日曆選日期**  
   在預約區塊先顯示**空房日曆**（與 availability 同源：`getBookedDates` 已訂／待確認／可訂），客人點選**入住日**、再點選**退房日**（或點兩次完成區間），系統自動帶入 `checkIn`、`checkOut`。
2. **第二步：房型與聯絡**  
   日期確定後，再填包棟規模（3／4／5 房）、加床、姓名、電話等；表單內不再手動輸入日期，避免選到已滿或不符合規則的區間。

**實作要點**：
- 前台預約區塊改為「先日曆、後表單」：同一頁或分步皆可；日曆可沿用 availability 的邏輯（月曆＋已訂／待確認／可訂標示），加上「點選起訖日」寫入 state，再帶入既有送出邏輯。
- API 不需改：仍用現有 `getBookedDates`、`createBooking`；僅前端流程與 UI 調整。
- 多房型／多間數產品若未來才上線，可維持現有表單式；包棟入口以日曆勾選為主，較符合「選時段就訂」的使用情境。

## 第八部分：部署架構與備份策略（規劃）

### 一、雲端分工（短期目標）

- **前端（畫面）**：Cloudflare Pages  
  - 本 repo 連到 Cloudflare，指定 `website/` 為輸出目錄。  
  - 網址規劃示例：`/`（官網）、`/agreement`、`/travel-guide`、`/agency`。

  - 保留現有 `doGet` / `doPost` API、Email 觸發器、旅遊手冊 7 天前、日曆同步等。  
  - Cloudflare 前端透過設定檔（如 `config.public.js`）呼叫 GAS Web App URL。

### 二、GAS 上線檢查清單

每次 `clasp push` 前，確認：

1. **必要檔案**：`main.js`、`schemaManager.js`、`dataStore.js`、`setup.js`、`EmailService.js`、`EmailTemplates.js`、`travelGuideSender.js`、`reminders.js`、`calendarSync.js` 等都不在 `.claspignore` 內。
2. **僅 GAS 需要的 HTML**：目前主要是 `admin.html`、`housekeeping.html`、`availability.html`（若仍在 GAS 上）。  
   - 官網相關頁面一律在 `website/`，預設給 Cloudflare / 靜態主機用，不強制 push 到 GAS。
3. **Script Properties**：包含試算表 ID、日曆 ID、寄件帳號等，與本文件描述一致。

### 三、備份策略

1. **主資料庫＝Google Sheets**  
   - 所有訂單與成本仍以 `訂單_YYYY`、`支出_YYYY` 等工作表為唯一欄位定義來源（由 `schemaManager.js` 管理）。
2. **冷備份（每日或每週）**  
   - 由 GAS 提供一個排程函式（之後實作）：  
     - 將當年度的 `訂單_YYYY`、`支出_YYYY` 匯出為 CSV。  
     - 上傳到 Google Drive 指定資料夾（如 `Backups/Orders_YYYY-MM-DD.csv`），或 push 到 GitHub 私有 repo。  
   - 用途：試算表誤刪或損毀時，仍可從 CSV 重建。
3. **熱備份（未來）**  
   - 規劃在 Cloudflare Workers 或外部 Node 服務上建一個 `/sync/order` 類 API。  
   - 在 `DataStore` 或建立／更新訂單邏輯內，於寫回 Sheets 後，同步呼叫該 API，將同一筆訂單寫入外部資料庫（例如 Postgres / Supabase / Cloudflare D1）。  
   - 之後若要替換後端，只需指向此外部資料庫即可，Sheets 仍可當報表與人工調整界面。

### 四、Cloudflare Pages 快速設定指南（摘要）

1. 在 Cloudflare 建立專案：選「Pages」→「Connect to Git」→ 選本 repo。  
2. Build 設定：  
   - Build command：留空（純靜態）。  
   - Build output directory：`website`。  
3. 若需要前端呼叫 GAS：在 Pages 專案的「環境變數」中設定 `API_BASE_URL`（值為目前的 GAS Web App URL），前端 JS 讀取這個變數。  
4. 自訂網域：將你的正式網域 CNAME 指向 Cloudflare 提供的 Pages 子網域，依介面指示完成驗證即可。
