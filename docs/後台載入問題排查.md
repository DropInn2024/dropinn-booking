# 後台一直「載入中」／config.js 404 排查

---

## 兩個 index 是什麼？

| 檔案 | 用途 |
|------|------|
| **根目錄 `index.html`** | 入口轉址用。打開網站根網址（例如 `https://dropinn.tw/`）時會執行這段，依路徑轉到：`/` → 訂房首頁、`/admin` → admin.html、`/housekeeping` → housekeeping.html。 |
| **`website/index.html`** | 真正的**訂房首頁**（表單、日曆、說明等）。 |

所以不是重複的兩個首頁，而是一個「路由器」+ 一個「訂房頁」。

---

## 後台「沒辦法登入」／一直載入、未授權

**可能原因與對策：**

1. **從靜態站（GitHub Pages／Cloudflare）開後台**，且 **config.public.js 的 `ADMIN_API_KEY` 是空的**  
   - 後端若在 GAS 有設 `ADMIN_API_KEY`，會回傳「未授權」。
   - **作法一**：在 `config.public.js` 裡把 `ADMIN_API_KEY` 填成和 GAS「專案設定 → 指令碼屬性」裡一樣的字串，存檔後重新整理後台。
   - **作法二（建議）**：不要從靜態站開後台，改從 **GAS 部署網址** 開：  
     `你的 Admin 部署網址?page=admin`  
     這樣頁面由 GAS 提供，會自動帶入金鑰，不用再輸入。

2. **`API_URL_ADMIN` 填錯或過期**  
   - 若你重新部署過 GAS，網址會變。請到 GAS「部署 → 管理部署作業」複製 **Admin API** 的「網址」，貼回 `config.public.js` 的 `API_URL_ADMIN`，然後 `git push` 更新靜態站。

3. **有跳出「需要後台通關碼」卻不知道密碼**  
   - 通關碼 = GAS 指令碼屬性裡的 `ADMIN_API_KEY`。若忘記，到 GAS 專案「專案設定 → 指令碼屬性」查看或重設；重設後記得把 `config.public.js` 的 `ADMIN_API_KEY` 一併更新（若你用靜態站開後台）。

## 步驟 1：確認 GAS 上的 admin.html 是否已是新版

在專案目錄執行（會把 GAS 目前的檔案拉回本機，**會覆蓋**你本機的 .gs/.js/.html）：

```bash
clasp pull
```

拉完後執行：

```bash
grep -n "config.js\|configJson" admin.html | head -5
```

- **若看到 `configJson`、沒有 `config.js`**：代表 GAS 上已是新版，問題在「部署用的版本」或瀏覽器快取（做步驟 2、3）。
- **若看到 `config.js`**：代表 GAS 上還是舊檔，可能是 push 到錯的專案或沒 push 成功，再執行一次 `clasp push` 後重做步驟 1。

---

## 步驟 2：確認你開的是「Admin API」且版本是最新

1. 打開 [Google Apps Script](https://script.google.com/) → 進入你的訂房系統專案。
2. 上方 **部署** → **管理部署作業**。
3. 在 **Admin API** 那一列記下：
   - **部署網址**（複製起來）。
   - **版本**：若是數字（例如 28），記下號碼；若有 **「標頭」/ Head**，代表用最新程式碼。
4. 點 Admin API 右側 **鉛筆（編輯）**：
   - **版本** 選 **「標頭」** 或 **數字最大的那一版** → 儲存。
5. 用**剛複製的網址** + `?page=admin` 開後台（不要用書籤或舊分頁）。

---

## 步驟 3：確認瀏覽器拿到的是新版 HTML

1. 用 **無痕視窗** 開後台網址：`你的Admin網址?page=admin`。
2. 在頁面按 **右鍵 → 檢視網頁原始碼**（或 Ctrl+U / Cmd+Option+U）。
3. 在原始碼裡 **搜尋**（Ctrl+F / Cmd+F）：
   - 搜 **`config.js`**：若**有找到**，代表目前這個部署還在送舊版，回到步驟 2 確認版本並儲存後再試。
   - 搜 **`configJson`** 或 **`FRONTEND_CONFIG`**：若**有找到**且沒有 config.js，代表送的是新版，若還是載入中，可能是後端 API（getAllOrders）錯誤，看步驟 4。

---

## 步驟 4：看主控台／網路錯誤

1. 開後台頁 → 按 **F12** → 選 **Console**。
2. 看有沒有**紅色錯誤**（例如 `FRONTEND_CONFIG is not defined`、`Unexpected token`、或 CORS／網路錯誤）。
3. 選 **Network**，重新整理，看 **Fetch/XHR** 裡有沒有請求失敗（紅色或 4xx／5xx），點進去看回應內容。

把錯誤訊息或失敗的請求網址記下來，方便進一步排查。

## 「儲存變更」出現「儲存失敗，請重試」

1. **確認部署權限**：Admin API 部署需為 **「誰可以存取：任何人」**。若為「僅限自己」，從 iframe 發出的 POST 可能被導向登入頁，回傳 HTML 導致前端判定失敗。
2. **看瀏覽器主控台**：按 F12 → Console。若出現「儲存失敗 HTTP: 403」或「回傳非 JSON」，代表後端回傳了錯誤頁面，請改為「任何人」後重新部署並再試。
3. **看 GAS 執行記錄**：在 Apps Script 編輯器 → **執行作業**／**紀錄**，查看 `doPost` 或 `updateOrderAndSync` 是否有錯誤（例如試算表權限、日曆 ID 未設定）。

## 「回傳格式錯誤」／「後端回傳資料格式錯誤」排查（2026-03 修正）

這個錯誤通常是前端在 `JSON.parse(...)` 時，拿到的不是 JSON，而是一整段 HTML 頁面（例如 Google 的包裝頁或登入頁）。

### 典型徵狀

- 後台進得去，但一載入就跳：
  - 「回傳格式錯誤」
  - 或 alert 顯示「後端回傳資料格式錯誤，請聯繫管理者協助查看 Logs。」
- 用 F12 → Network 看 `?action=getAllOrders`，會發現其實拿到的是一整個 HTML。

### 根本原因（這次實際發生的狀況）

- `admin.html` 裡的 `API_URL` 曾被改成：

  ```js
  const API_URL = window.location.origin + window.location.pathname;
  ```

- 當後台是經由 `script.google.com/.../userCodeAppPanel?...` 這種「外層包裝頁」開啟時：
  - `window.location.origin + window.location.pathname` 指向的是 **包裝頁本身**，
  - 不是 `.../macros/s/XXXX/exec` 的真正 GAS Web App API。
- 結果 `fetch(API_URL + '?action=getAllOrders&adminKey=...)` 打到的是包裝頁，後端回傳整個 HTML，前端 `JSON.parse` 就爆掉。

### 正確作法（已在程式中修正）

- 改為使用 GAS 從後端注入的正式 API 網址（`ScriptApp.getService().getUrl()`），由 `configLoader.gs` 組成 `configJson` 並塞進 `FRONTEND_CONFIG`：

  ```js
  // admin.html（全域變數區）
  let allOrders = [];
  let currentOrder = null;
  // 使用 GAS 注入的正式 API 網址（exec），避免誤打到外層 userCodeAppPanel 頁面
  const API_URL = FRONTEND_CONFIG.API_URL;
  const ADMIN_API_KEY = FRONTEND_CONFIG.ADMIN_API_KEY || '';
  ```

- 之後所有：
  - `fetch(API_URL + '?action=getAllOrders&...')`
  - `fetch(API_URL, { method: 'POST', body: ... })`

  都會直接打到真正的 `/exec` API，而不是外層 HTML。

### 快速檢查方式

1. 在後台頁面按 F12 → Console，輸入：

   ```js
   FRONTEND_CONFIG && FRONTEND_CONFIG.API_URL
   ```

   - 應該要看到一個 `https://script.google.com/macros/s/XXXX/exec` 類型的網址。

2. 在 `admin.html` 搜尋 `API_URL`，確認是：

   ```js
   const API_URL = FRONTEND_CONFIG.API_URL;
   ```

   而不是 `window.location.origin + window.location.pathname`。

3. Network 面板點 `?action=getAllOrders...`，Response 應該是 JSON 陣列或 `{ success: false, error: ... }`，而不是 `<html>...`。

## housekeeping.html 載入失敗的根本原因紀錄（2026-03）

### 問題現象

- 從後台或 iframe 開啟 `housekeeping.html` 時，房務頁常常整頁掛掉或一直載入中。
- 主控台可看到 CORS 錯誤或 fetch 失敗，但 admin 後台同一時間是正常的。

### 根本原因

- **架構不一致**：
  - `admin.html` 採用 **雙模式**：
    - GAS 環境：優先用 `google.script.run.xxx()` 呼叫後端。
    - 靜態部署（GitHub Pages / Cloudflare）：用 `fetch(API_URL)` 當 fallback。
  - 但 `housekeeping.html` 一直是**純 fetch**，在 GAS iframe（`userCodeAppPanel`）環境裡：
    - 這種「外層包裝頁」對跨來源請求有保護，直接從 iframe 發出的 `fetch` 容易被 CORS 擋掉。
    - 結果導致房務頁在 GAS 部署環境下常常取不到資料或整頁錯誤。

- **兩個頁面靠不同邏輯決定 API_URL**：
  - admin 早已統一用 GAS 注入的 `FRONTEND_CONFIG.API_URL_ADMIN` / `API_URL`。
  - housekeeping 曾經直接組 `window.location` 或硬塞網址，和 admin 邏輯不同，之後只要改其中一邊，很容易再次不同步。

### 已採取的修正（2026-03）

1. **housekeeping.html 與 admin.html 對齊架構**
   - 新版 `housekeeping.html` 也改成：
     - 優先使用 `google.script.run.adminGetAllOrders()` 從 GAS 取得資料。
     - 非 GAS 環境（例如純靜態站）時，才透過 fetch fallback（由 `ensureFetchBackedGoogleScriptRun_()` 模擬 `google.script.run`）。
   - 這樣在 GAS iframe 裡，請求是從 GAS 端發出，不會再被瀏覽器 CORS 擋掉。

2. **統一 API 設定來源**
   - `API_URL`、`ADMIN_API_KEY` 一律從 `FRONTEND_CONFIG` 讀取：
     - `FRONTEND_CONFIG.API_URL_ADMIN || FRONTEND_CONFIG.API_URL`。
   - `FRONTEND_CONFIG` 則由：
     - GAS 端 `configLoader.gs` 注入（GAS 部署）。
     - 或前台 `config.public.js` 提供（靜態部署）。
   - `housekeeping.html` 不再自己亂組 `window.location.origin + pathname` 這種網址。

3. **加入安全防護與友善錯誤畫面**
   - 若 `API_URL` 未設定或過短：
     - 不再送出請求，而是在畫面中顯示明確訊息：「API 網址未設定，請從後台連結開啟房務頁或檢查 config.public.js」。
   - 若後端回傳 `{ success: false, error: '...' }` 或回應格式錯誤：
     - 不再被當成訂單陣列處理，而是顯示「後端回傳錯誤」＋具體錯誤訊息。

### 之後修改 API 呼叫方式的原則

- **任何新頁面（admin / housekeeping / availability / website 下其他管理頁）若要打 Admin API：**
  1. **一律載入官方共用 helper**：`<script src="dropinn-api-helper.js"></script>`（在 `config.public.js` 之後），並在 `window.onload` 呼叫一次 `dropinnEnsureRuntime()`。**禁止在頁面內自己組 fetch 或重寫 google.script.run。**
  2. 若尚未改用 helper，則沿用 `admin.html` 的雙模式架構（GAS 用 `google.script.run`，靜態用 fetch fallback）；API 網址與金鑰從 `FRONTEND_CONFIG` 讀取。
- 總結：**新頁面打 Admin API 一律用 dropinn-api-helper.js，禁止自己組 fetch。**

## 上線前建議與已知事項（2026-03）

- **config.public.js 的 ADMIN_API_KEY**：從靜態站開後台／房務頁時，必須與 GAS「指令碼屬性」的 `ADMIN_API_KEY` 一致，否則會被擋。若留空，請改從 GAS 部署網址 `?page=admin` / `?page=housekeeping` 開啟。
- **getAllOrders 已加金鑰保護**：`main.js` 的 `isAdminAction` 已包含 `getAllOrders`，任何人 POST 該 action 都須帶正確 `adminKey`。
- **admin.html 已讀取 GAS 注入**：GAS 部署時由 `main.js` 注入的 `configJson`（API 網址與金鑰）已由 admin 頁面讀取並寫入 `FRONTEND_CONFIG`，從 GAS 開後台會自動帶入金鑰。
- **訂房表單（website/index.html）**：若目前是「客人填表後由你手動建單」則無需改動；若要真正線上接單，需補上打 GAS API 建立訂單與 reCAPTCHA。
- **housekeeping 在 GAS 不注入 API URL**：房務頁在 GAS 環境只靠原生 `google.script.run`，不注入 config，屬刻意設計；靜態站則靠 `config.public.js` 的 `API_URL_ADMIN`。
- **housekeeping/index.html、admin/index.html 的 GAS 轉址 URL**：若未來重新部署 GAS（換 script ID），記得同步更新這兩個轉址頁裡的硬編碼網址。
