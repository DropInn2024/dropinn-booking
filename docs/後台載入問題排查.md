# 後台一直「載入中」／config.js 404 排查

## 步驟 1：確認 GAS 上的 admin.html 是否已是新版

在專案目錄執行（會把 GAS 目前的檔案拉回本機，**會覆蓋**你本機的 .gs/.js/.html）：

```bash
clasp pull
```

拉完後執行：

```bash
grep -n "config.js\|configJson" admin.html | head -5
```

- **若看到 `configJson`、沒有 `config.js`**：代表 GAS 上已是新版，問題在「部署用的版本」或瀏覽器快取（做步驟 2、3）。
- **若看到 `config.js`**：代表 GAS 上還是舊檔，可能是 push 到錯的專案或沒 push 成功，再執行一次 `clasp push` 後重做步驟 1。

---

## 步驟 2：確認你開的是「Admin API」且版本是最新

1. 打開 [Google Apps Script](https://script.google.com/) → 進入你的訂房系統專案。
2. 上方 **部署** → **管理部署作業**。
3. 在 **Admin API** 那一列記下：
   - **部署網址**（複製起來）。
   - **版本**：若是數字（例如 28），記下號碼；若有 **「標頭」/ Head**，代表用最新程式碼。
4. 點 Admin API 右側 **鉛筆（編輯）**：
   - **版本** 選 **「標頭」** 或 **數字最大的那一版** → 儲存。
5. 用**剛複製的網址** + `?page=admin` 開後台（不要用書籤或舊分頁）。

---

## 步驟 3：確認瀏覽器拿到的是新版 HTML

1. 用 **無痕視窗** 開後台網址：`你的Admin網址?page=admin`。
2. 在頁面按 **右鍵 → 檢視網頁原始碼**（或 Ctrl+U / Cmd+Option+U）。
3. 在原始碼裡 **搜尋**（Ctrl+F / Cmd+F）：
   - 搜 **`config.js`**：若**有找到**，代表目前這個部署還在送舊版，回到步驟 2 確認版本並儲存後再試。
   - 搜 **`configJson`** 或 **`FRONTEND_CONFIG`**：若**有找到**且沒有 config.js，代表送的是新版，若還是載入中，可能是後端 API（getAllOrders）錯誤，看步驟 4。

---

## 步驟 4：看主控台／網路錯誤

1. 開後台頁 → 按 **F12** → 選 **Console**。
2. 看有沒有**紅色錯誤**（例如 `FRONTEND_CONFIG is not defined`、`Unexpected token`、或 CORS／網路錯誤）。
3. 選 **Network**，重新整理，看 **Fetch/XHR** 裡有沒有請求失敗（紅色或 4xx／5xx），點進去看回應內容。

把錯誤訊息或失敗的請求網址記下來，方便進一步排查。

## 「儲存變更」出現「儲存失敗，請重試」

1. **確認部署權限**：Admin API 部署需為 **「誰可以存取：任何人」**。若為「僅限自己」，從 iframe 發出的 POST 可能被導向登入頁，回傳 HTML 導致前端判定失敗。
2. **看瀏覽器主控台**：按 F12 → Console。若出現「儲存失敗 HTTP: 403」或「回傳非 JSON」，代表後端回傳了錯誤頁面，請改為「任何人」後重新部署並再試。
3. **看 GAS 執行記錄**：在 Apps Script 編輯器 → **執行作業**／**紀錄**，查看 `doPost` 或 `updateOrderAndSync` 是否有錯誤（例如試算表權限、日曆 ID 未設定）。

## 「回傳格式錯誤」／「後端回傳資料格式錯誤」排查（2026-03 修正）

這個錯誤通常是前端在 `JSON.parse(...)` 時，拿到的不是 JSON，而是一整段 HTML 頁面（例如 Google 的包裝頁或登入頁）。

### 典型徵狀

- 後台進得去，但一載入就跳：
  - 「回傳格式錯誤」
  - 或 alert 顯示「後端回傳資料格式錯誤，請聯繫管理者協助查看 Logs。」
- 用 F12 → Network 看 `?action=getAllOrders`，會發現其實拿到的是一整個 HTML。

### 根本原因（這次實際發生的狀況）

- `admin.html` 裡的 `API_URL` 曾被改成：

  ```js
  const API_URL = window.location.origin + window.location.pathname;
  ```

- 當後台是經由 `script.google.com/.../userCodeAppPanel?...` 這種「外層包裝頁」開啟時：
  - `window.location.origin + window.location.pathname` 指向的是 **包裝頁本身**，
  - 不是 `.../macros/s/XXXX/exec` 的真正 GAS Web App API。
- 結果 `fetch(API_URL + '?action=getAllOrders&adminKey=...)` 打到的是包裝頁，後端回傳整個 HTML，前端 `JSON.parse` 就爆掉。

### 正確作法（已在程式中修正）

- 改為使用 GAS 從後端注入的正式 API 網址（`ScriptApp.getService().getUrl()`），由 `configLoader.gs` 組成 `configJson` 並塞進 `FRONTEND_CONFIG`：

  ```js
  // admin.html（全域變數區）
  let allOrders = [];
  let currentOrder = null;
  // 使用 GAS 注入的正式 API 網址（exec），避免誤打到外層 userCodeAppPanel 頁面
  const API_URL = FRONTEND_CONFIG.API_URL;
  const ADMIN_API_KEY = FRONTEND_CONFIG.ADMIN_API_KEY || '';
  ```

- 之後所有：
  - `fetch(API_URL + '?action=getAllOrders&...')`
  - `fetch(API_URL, { method: 'POST', body: ... })`

  都會直接打到真正的 `/exec` API，而不是外層 HTML。

### 快速檢查方式

1. 在後台頁面按 F12 → Console，輸入：

   ```js
   FRONTEND_CONFIG && FRONTEND_CONFIG.API_URL
   ```

   - 應該要看到一個 `https://script.google.com/macros/s/XXXX/exec` 類型的網址。

2. 在 `admin.html` 搜尋 `API_URL`，確認是：

   ```js
   const API_URL = FRONTEND_CONFIG.API_URL;
   ```

   而不是 `window.location.origin + window.location.pathname`。

3. Network 面板點 `?action=getAllOrders...`，Response 應該是 JSON 陣列或 `{ success: false, error: ... }`，而不是 `<html>...`。
